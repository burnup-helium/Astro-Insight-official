# Astro-Insight 天文数据可视化平台项目提案
## 1. 项目一句话定位
Astro-Insight 是一个面向高维天文数据的探索性可视化分析平台，目标不是单纯展示天文结果，而是帮助用户理解数据结构、比较不同可视化策略对认知的影响，并追踪从数据清洗、EDA、降维、联动分析到解释性结论生成的完整过程。
## 2. 项目目标
1. 支持天文数据的探索性分析与结构发现。
2. 让用户看到不同可视化策略如何改变认知结果。
3. 将平台从“结果展示”升级为“信息获取过程展示”。
4. 保留现有项目的数据、分析与工程基础，在此之上升级前端交互与可视化叙事。
5. 建立一个既有 MIT 式的清晰、锋利、任务导向，又有 Stanford 式的博学、解释充分、方法比较完整的分析工作台。
## 3. 核心故事线
项目主线是：
> 同一份天文数据，在不同可视化选择、不同预处理方式、不同降维方法与不同参数下，会呈现不同的结构、差异与洞察。平台的任务不是给出唯一答案，而是展示“答案如何被发现”。
这意味着平台要强调：
- 结构如何被发现
- 异常如何被定位
- 差异如何被比较
- 结论如何被解释
- 可视化如何塑造认知

## 4. 数据策略
### 4.1 核心数据方向
优先接入 NASA Exoplanet Archive 的数据，并保留现有项目里已经存在的天体分类、检索、统计分析基础。该数据源天然适合本项目，因为它具有：
- 高维数值特征
- 可用于结构发现的多变量属性
- 可比较的样本规模
- 适合 PCA / t-SNE / UMAP 等方法
- 适合做任务导向的可视化推荐
### 4.2 数据分层策略
建议采用“本地核心集 + API 扩展集 + 结果缓存集”的三层结构：
| 层级 | 作用 | 数据内容 | 目的 |
|---|---|---|---|
| 本地核心集 | 快速分析与演示 | 视星等较低、常用、适合交互的天体样本 | 支持快速加载、联动、结构探索 |
| API 扩展集 | 按需补充 | NASA Exoplanet Archive 等外部数据源 | 保持样本规模与数据丰富度 |
| 结果缓存集 | 复用分析结果 | PCA/t-SNE/UMAP 坐标、聚类标签、图表配置、推荐结果 | 避免重复计算，支持对比与回溯 |
### 4.3 适合保留的数据特征
平台至少应支持以下类型：
- 数值型特征：质量、半径、温度、距离、周期、星等等
- 空间或分布信息：RA/Dec、天区分布、观测位置
- 多维特征：可用于降维和结构比较
- 方法对比信息：不同图表、不同算法、不同参数的结果
### 4.4 后续扩展方向
后续可加入：
- 晴天钟/夜观条件相关数据
- 天体升落时间相关数据
- 时序观测与周期性模式
这些内容不作为当前主线，而是骨架搭好后的扩展任务。

## 5. 数据库对象设计
数据库不只是存数据，而是要支撑分析流程、可视化流程和交互过程。建议至少包含以下对象：
| 对象 | 作用 | 说明 |
|---|---|---|
| 原始数据表 | 保留导入数据 | 保存未清洗、未变换前的数据 |
| 清洗后的数据表 | 存放整齐数据 | 记录缺失处理、异常处理、单位统一后的结果 |
| 特征表 | 支撑分析建模 | 保存标准化、特征工程、降维前的特征向量 |
| 可视化任务表 | 记录分析任务 | 存储任务类型、筛选条件、分析方法、状态 |
| 图表配置表 | 记录图表生成方式 | 存储图表类型、编码方式、主题、参数 |
| 用户会话表 | 记录分析上下文 | 存储会话、筛选状态、当前页面、交互轨迹 |
| 算法结果表 | 记录分析结果 | 存储 PCA、聚类、异常检测、相关性等结果 |
| 图表生成记录表 | 记录可视化产物 | 存储每次图表生成的参数与来源 |
| 交互事件表 | 记录用户行为 | 点击、筛选、缩放、切换、回退等行为日志 |
## 6. 页面与信息架构
平台应定位为“分析工作台”，而不是普通展示页。建议采用以下页面结构：
### 6.1 首页
目标：解释为什么这个数据需要可视化，以及这个平台能完成什么任务。
建议内容：
- 项目定位
- 数据来源说明
- 核心任务入口
- 方法流程图：数据 → 清洗 → EDA → 降维 → 对比 → 解释
- 平台特色：图表推荐、多视图联动、过程展示

### 6.2 数据概览页
目标：让用户快速理解数据结构。
建议内容：
- 字段列表
- 样本量
- 缺失情况
- 基础统计
- 分布情况
- 类别频次
- 数据来源与版本
### 6.3 分析页（EDA 工作台）
目标：完成筛选、聚合、排序、异常查看、相关性查看。
建议内容：
- 单变量分布
- 双变量关系
- 相关性矩阵
- 异常点查看
- 子集比较
- 联动筛选
### 6.4 结构页（降维与分型）
目标：展示 PCA / t-SNE / UMAP 的结构空间。
建议内容：
- 方法切换
- 参数调整


### 6.5 对比页
目标：比较不同图、不同方法、不同参数、不同特征集对理解结果的影响。
建议内容：
- PCA vs t-SNE vs UMAP
- 原始空间 vs 标准化后空间
- 不同图表类型对同一任务的表现
- 不同筛选条件下结构变化
- 不同参数下的认知差异
### 6.6 结果页
目标：输出解释性结论，而不是只给图。
建议内容：
- 哪种图适合什么任务
- 哪些变量驱动结构
- 哪些样本可能异常
- 哪种算法更适合当前数据
- 结论与证据链回链
### 6.7 行为分析页（建议新增）
这是整个平台的一个亮点扩展页，用于分析用户如何分析数据。
建议内容：
- 用户路径图
- 点击热力图
- 停留时间分布
- 操作序列分析
- 行为与结果关联分析
## 7. 交互设计原则
平台必须是交互式的，支持以下能力：
- 图表切换
- 视图联动
- 点击高亮
- 框选过滤
- 参数调整
- 结果对比
- 图表说明
- 状态回退
- 状态重置
建议交互风格是“分析工作台”而不是“结果海报”。用户一进入页面就应理解：这里是做探索、验证与解释的地方。
## 8. 可视化策略设计
平台的关键不是图越多越好，而是图与任务要匹配。
### 8.1 四类核心图
| 图类 | 目标 | 适合回答的问题 |
|---|---|---|
| 总览图 | 看规模与分布 | 数据整体长什么样、是否偏态、是否有缺失与离群 |
| 结构图 | 看空间与分型 | 数据是否存在簇、结构如何形成 |


### 8.2 可视化方法建议
- 总览：柱状图、直方图、箱线图、密度图、缺失矩阵
- 结构：PCA 散点图、t-SNE、UMAP、聚类图
- 对比：并排小 multiples、矩阵对比图、参数联动图
- 解释：热力图、贡献度图、相关性图、异常点表格
### 8.3 图表推荐逻辑
图表推荐不应只返回“一个图”，而应返回：
- 推荐图类型
- 推荐原因
- 适用任务
- 不推荐原因
- 替代方案
推荐依据分三层：
1. 任务驱动：结构、差异、关系、异常、分布
2. 数据驱动：变量类型、缺失率、样本量、相关性
3. 方法驱动：PCA、t-SNE、UMAP、热力图、箱线图等的适配性
## 9. 多视图联动设计
多视图联动是本项目的核心创新之一。建议实现以下链路：
- 点击样本点 → 其他视图同步高亮
- 框选样本簇 → 表格、统计、分布同步更新
- 调整参数 → 降维图和解释图同步刷新
- 改变筛选条件 → 所有视图统一更新


## 11. 用户历史行为分析
建议把“用户操作或会话表”升级为分析对象的一部分，记录用户如何探索数据。
### 可分析的行为包括：
- 点击
- hover
- 框选
- 缩放
- 过滤
- 排序
- 切图
- 参数调整
- 导出
- 重置
### 可视化行为结果包括：
- 用户路径图
- 行为热力图
- 任务转移图
- 停留时间分析
- 行为与结果关联图
这会让平台更有“分析认知过程可解释”的气质。

## 12. 数据与课程对齐
平台应明确对应课程内容：
- 数据预处理
- EDA 流程
- PCA / SVD
- 可视化编码与设计
- 多视图联动
- 信息可视化
- 分析结果解释
这也意味着页面、模块、接口都要围绕“探索性分析与可视化设计”展开，而不是偏离课程主题做成通用天文门户。


## 13. 设计气质
项目风格建议同时满足两种气质：
### MIT 式锋利
- 问题定义清晰
- 任务导向明确
- 交互链路短
- 计算结果可验证
- 每个视图都有明确作用
### Stanford 式博学
- 数据语境完整
- 方法比较充分
- 解释性强
- 结果有证据链
- 不同可视化策略背后的知识结构清晰
## 14. 核心创新点
1. 可视化策略选择：让系统根据数据特征和任务推荐图表。
2. 多视图联动：一个视图的操作驱动其他视图同步变化。
3. EDA 过程可解释：展示清洗前后、降维前后、不同方法对比。
4. 算法结果比较：强调不同算法为什么得出不同结构。
5. 信息任务导向：围绕结构、差异、关系、异常组织页面和图表。
6. 行为分析扩展：把用户如何探索数据也纳入可视化对象。
## 15. 里程碑计划
| 阶段 | 目标 | 交付物 |
|---|---|---|
| Phase 1 | 定义与收敛 | 项目说明、数据字典、页面原型、接口草案 |
| Phase 2 | 数据与后端 | 数据接入、数据库 schema、分析接口、结果缓存 |
| Phase 3 | 前端工作台 | 首页、概览、分析、结构、对比、结果页面 |
| Phase 4 | 方法比较与解释 | PCA/t-SNE/UMAP 对比、图表推荐、解释面板 |
| Phase 5 | 打磨与展示 | 视觉优化、交互收敛、项目演示版、文档 |
## 16. 风险与规避
| 风险 | 表现 | 规避策略 |
|---|---|---|
| 数据太散 | 主线被冲淡 | 只保留一条核心天文数据主线，其他作为扩展 |
| 图太多 | 用户认知负担高 | 每页只服务一个任务，所有图附解释 |
| 方法秀化 | 算法比较变成展示算法 | 强调方法与任务的对应关系 |
| 联动混乱 | 交互状态不统一 | 建立统一状态管理与回退机制 |
| 结果空泛 | 只有图没有结论 | 所有结论必须回链到图、数据和方法 |
## 17. 建议的最终项目定义
> 本项目是一个面向高维天文数据的探索性可视化工作台，支持数据清洗、特征构建、降维分析、多视图联动、图表推荐与用户行为分析，用于研究不同可视化策略如何影响对天文结构的理解。
## 18. 当前最推荐的落地顺序
1. 确定 NASA Exoplanet Archive 的核心字段范围
2. 定义本地核心集与 API 扩展集策略
3. 设计数据库 schema
4. 设计页面信息架构
5. 设计统一 API 接口
6. 先做 EDA + PCA + 对比页
7. 再做多视图联动与图表推荐
8. 最后补行为分析与解释性结论页
