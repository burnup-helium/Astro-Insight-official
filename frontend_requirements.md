项目定位

这是一个基于你们原 Astro-Insight 原型继续扩展的“天文数据可视化分析平台”。
核心不是单纯展示图，而是让用户能理解高维天文数据的结构、差异和可视化方法选择。

核心目标

1. 让用户能完成天文数据的探索性分析。
2. 让用户能看到不同可视化策略对理解结果的影响。
3. 让平台从“结果展示”升级为“信息获取过程展示”。
4. 保留你们原项目的数据和分析基础，但把重点转到可视化与前端交互。

你这次平台最重要的创新点

可视化策略选择。
系统根据数据特征和分析任务，推荐合适的图表类型。

多视图联动。
一个视图的筛选、点击、缩放，能驱动其他视图同步变化。

EDA过程可解释。
展示清洗前后、降维前后、不同方法对比，让用户看到结构是怎么被发现的。

算法结果比较。
比如 PCA、t-SNE、UMAP 之类方法的结构差异对比，重点是解释“为什么图会不一样”。

信息任务导向。
围绕结构、差异、关系、异常这几类任务来组织可视化，而不是堆图。

数据与内容要求

你最好保留“高维、多变量、适合结构分析”的天文数据。
平台里至少要有一类数据能支持这些内容：

* 数值型特征
* 可能的空间或分布信息
* 可用于降维的多维特征
* 可用于比较不同视图的结果

如果原项目里已经有天体分类、检索、统计分析数据，就优先沿用，不必为了扩展去追求完全新的数据源。

页面与功能要求

首页要能说明项目背景和任务。
不是长篇介绍，而是清楚说明“为什么这个数据需要可视化”。

数据概览页要能看见数据结构。
包括字段、样本量、缺失情况、基础统计、分布情况。

分析页要能做 EDA。
包括筛选、聚合、排序、异常查看、相关性查看。

结构页要能做降维和分型。
这是你项目的重点页面，展示 PCA 或其他降维后的结构空间。

对比页要能比较不同方法。
同一数据用不同图、不同算法、不同参数呈现，比较信息获取效果。

结果页要能给出解释性结论。
不是只给图，而是给“哪种图更适合什么任务”的结论。

数据库要求

数据库不只是存数据，要能支撑分析流程。
你需要至少设计出这几类对象：

* 原始数据表
* 清洗后的数据表
* 特征表
* 可视化任务表
* 图表配置表
* 用户操作或会话表

如果你要做得更完整，还可以把“算法结果”和“图表生成记录”也存进去，这样平台会更像一个真实系统。

后端要求

后端要负责三件事：

1. 数据读取和清洗。
2. EDA与分析计算。
3. 为前端提供统一接口。

后端接口最好覆盖这些能力：

* 获取数据概览
* 获取某个分析任务的结果
* 获取降维后的坐标
* 获取图表推荐结果
* 获取不同方法的对比结果
* 获取当前筛选条件下的联动数据

前端要求

前端必须是交互式的。
不能只是静态图页面。

至少要支持：

* 图表切换
* 视图联动
* 点击高亮
* 筛选过滤
* 参数调整
* 结果对比
* 图表说明

前端风格上要偏“分析工作台”，不是普通展示页。
也就是用户一进来就能知道：这里是用来探索数据的。

可视化要求

你这次要重点做的不是“图越多越好”，而是“图之间有逻辑”。

建议保留四类图：

1. 总览图。
   看分布、规模、基本结构。

2. 结构图。
   看 PCA、嵌入空间、聚类。

3. 对比图。
   看不同方法、不同参数、不同任务下的差异。

4. 解释图。
   看变量贡献、相关性、异常点、结论说明。

课程对齐要求

这套平台必须能明确对应课程内容：

* 数据预处理
* EDA流程
* PCA 或 SVD
* 可视化编码与设计
* 多视图联动
* 信息可视化
* 分析结果解释

也就是说，老师看到后能直接判断：
这是一个围绕“探索性分析和可视化设计”搭起来的项目。

---

- 目前有的项目是后端的天文研究工作流
- 主入口说明主要在 `README.md`
- 真正的前端主页面是 `templates/index.html`
- 前端逻辑在 `static/js/main.js`
- 现有前端更像“API 演示 + 轻量后台面板”，不是完整分析工作台

我对项目的判断是：

1. 入口
   - 论文式/说明式入口：`README.md`
   - Web 页面入口：`templates/index.html`
   - 交互逻辑入口：`static/js/main.js`
   - API 示例页：`api_service/frontend_examples.html`

2. 作用
   - 面向天文问答、天体检索、数据分析、分类、代码生成
   - 后端靠 LangGraph + MCP 组织流程
   - 前端只负责发起分析、展示结果、展示统计

3. 现有前端是干什么的
   - 主页有 3 个区：分析 / 数据 / 统计
   - 分析页：输入问题、选择用户类型和分析类型，然后调用接口拿结果
   - 数据页：看对象数、分类数、执行历史，并用表格看明细
   - 统计页：用 Chart.js 看分析类型分布、用户类型分布、成功率趋势
   - 所以它本质上是“结果查看器 + 简单控制台”，不是一个真正的分析界面

the design must be able to answer:
不同可视化方法会不会让人对同一组天文数据产生不同理解？
我们看到的“天文结构”，到底是数据本身的，还是可视化方法制造的？
